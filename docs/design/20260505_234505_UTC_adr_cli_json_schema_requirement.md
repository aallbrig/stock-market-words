# ADR: Every CLI-generated JSON file must have a JSON Schema

**Status:** Accepted  
**Date:** 2026-05-05  
**Deciders:** Andrew Allbright

---

## Context

The `ticker-cli` pipeline generates multiple JSON files consumed by the Hugo static site and browser JavaScript. As the number of output files grows (currently eight: `all_tickers.json`, six `strategy_*.json` files, `filtered_tickers.json`, `pass1_results.json`, `raw_nasdaq.json`, `raw_otherlisted.json`, `ticker-lookup.json`, `trie.json`, `metadata.json`), it becomes easy for:

- A code change to silently alter a field name, type, or nesting depth that downstream templates and JS depend on.
- A new output file to be added without documenting its contract.
- Runtime bugs in the Hugo build or browser JS to be traced back only after a deploy, rather than caught at generation time.

We have no automated check between "what Python writes" and "what Hugo/JS reads."

---

## Decision

**Any JSON file produced by `ticker-cli` must have a corresponding JSON Schema (Draft-07) committed to `hugo/site/static/schemas/`.**

The schema filename mirrors the output filename with a `.schema.json` suffix (e.g., `all_tickers.json` → `all_tickers.schema.json`). When one schema covers a family of files with the same shape (e.g., all six `strategy_*.json` files), a single `strategy.schema.json` is sufficient with the covered filenames noted in the schema's `description` field.

This rule applies at authoring time: a PR that adds or modifies a CLI-generated JSON output is not complete until the schema is added or updated in the same commit.

---

## Consequences

### Positive

- **Contracts are explicit and version-controlled.** Hugo templates and JS can be written against the schema rather than inferred behavior.
- **CI can validate outputs.** Running `ticker-cli run-all` in CI and then validating every generated JSON file against its schema catches regressions before deploy.
- **Schemas serve as living documentation.** Field names, types, nullable flags, and value ranges are stated once in a machine-readable form rather than scattered across comments.
- **Schemas are browsable at runtime.** Because they live in `hugo/site/static/schemas/`, they are served at `/schemas/*.schema.json` and can be linked from the site or used by external consumers.

### Negative / trade-offs

- **Small authoring overhead.** Every new output file requires a schema to be written. Acceptable cost given the stability benefits.
- **Schemas can drift.** A schema is only useful if it stays in sync with the generator. The CI validation step (see enforcement below) is what keeps this honest — the schema is the test.

---

## Enforcement

Two layers are expected to enforce this over time:

1. **CI validation (to be implemented):** After `ticker-cli run-all`, a validation step runs `jsonschema` (Python) or equivalent against every generated file. A mismatch fails the build. This is the primary safety net.

2. **PR convention:** The schema file is part of the definition of done for any PR that touches a JSON generator. Reviewers should treat a missing or unstated schema update as a blocking issue.

---

## Scope

This decision covers files generated by the CLI pipeline (`hugo_generators.py`, `builders.py`, and any future generator modules). It does not cover:

- Hand-authored JSON files (e.g., `static/schemas/models/navigation.json`).
- JSON embedded inside Hugo templates or JavaScript source.
- Intermediate or temporary files that are never committed or deployed.

---

## Current state

As of this ADR, schemas exist for all current CLI outputs:

| Schema file | Covers |
|---|---|
| `all_tickers.schema.json` | `data/all_tickers.json` |
| `strategy.schema.json` | `static/data/strategy_*.json` (6 files) |
| `filtered_tickers.schema.json` | `static/data/filtered_tickers.json` |
| `pass1_results.schema.json` | `static/data/pass1_results.json` |
| `raw_ftp.schema.json` | `static/data/raw_nasdaq.json`, `static/data/raw_otherlisted.json` |
| `ticker_lookup.schema.json` | `static/data/ticker-lookup.json` |
| `trie.schema.json` | `static/api/trie.json` |
| `metadata.schema.json` | `static/api/metadata.json` |

The CI validation step is not yet implemented. That is the highest-priority follow-on from this decision.
